# Cursor Rule（プロジェクト開発ルール）

1. 明確なルール設定: プロジェクト開始時に5〜10個のルールを策定し、構造や制約をCursorに共有する。既存コードベースでは`/generate rules`も活用する。  
2. 具体的なプロンプト作成: 技術スタックや期待挙動、制約条件をミニ仕様書として明確に記述する。  
3. ファイル単位で作業: 小さな単位（ファイルごと）で生成・テスト・レビューを繰り返し、段階的に品質を高める。  
4. テスト駆動開発（TDD）: まずテストを書き、すべてのテストが通るまでコード生成・修正を続ける。  
5. AI出力のレビューと学習: AI出力は必ず人間がレビューし、問題があれば自分で修正し、その修正例をCursorに学習させる。  
6. 適切なフォーカス指定: `@ file`、`@ folders`、`@ git` などを活用し、Cursorが適切な箇所にフォーカスできるようにする。  
7. 設計文書・チェックリスト管理: 設計文書やチェックリストは`.cursor/`に保存し、次作業者が完全に理解できるようにする。  
8. 自己修正の徹底: コードに誤りがあれば説明より先に自分で修正し、その修正をCursorに学習させる。  
9. チャット履歴の活用: 過去のプロンプトややりとりを活用し、ゼロからではなく改善しながら再利用する。  
10. モデル選択の意図明確化: 目的に応じてモデルを選択する（例: 精度重視はGemini、広範囲の知識はClaude）。  
11. 新技術への対応: 新技術や未経験スタックでは公式ドキュメントのリンクを提示し、エラーや修正箇所を1行ずつ説明させる。  
12. 大規模プロジェクトのインデックス化: 大規模プロジェクトは一晩かけてインデックス化し、パフォーマンスのためコンテキスト範囲を制限する。
